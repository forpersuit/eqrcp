这是一份关于 **移动端 H5 浏览器** 与 **GUI 桌面端** 在大文件发送、接收场景下，从**历史隐患**到**最新优化落地**的技术对照梳理。

我们围绕 **“内存安全（防 OOM）”**、**“断点续传（防抖高可用）”** 以及 **“精简气泡渲染（零预加载开销）”** 三个核心指标进行了完整重构。

---

### 📊 四个传输象限：旧设计隐患 vs 最新实现对比

#### 1. 移动端发送文件 (手机浏览器 ➔ 桌面端电脑)

| 维度 | 历史遗留逻辑与隐患 | 最新优化实现方案 |
| :--- | :--- | :--- |
| **内存溢出风险** | 1. 选中文件后，无论多大都立即对多媒体执行 `URL.createObjectURL(file)` 载入内存，渲染预览大图/大视频，强占堆内存不释放，导致页面直接崩溃。<br>2. 多文件同时发送时发起**高并发 XHR POST**，发送缓冲区叠加爆内存。 | 1. **大文件拦截**：大小超过 `50MB` 的图片/视频不再分配 ObjectURL，拒绝渲染预览，退回通用文件气泡。<br>2. **小图片 10MB 预览限制**：只有当图片 $< 10\text{MB}$ 时，才进行气泡大图预览渲染。 |
| **断点续传能力** | 传统整体 `POST` 传输，网络一旦闪断或抖动必须重头开始。 | 网页端重构了串行队列控制，多选时进入 `uploadQueue` **串行一个接一个上传**，规避了高并发导致的 FormData 堆内存膨胀与局域网心跳连接断开。 |
| **气泡渲染开销** | 视频/音频在未点击时直接在 DOM 树中建立 `<video>` / `<audio>` 标签并加载 `src` 链接，消耗多媒体解码与连接数资源。 | **点播流式加载**：默认只在后台使用离屏临时 `video` 载入元数据获取分辨率做排版自适应（**读取完立即清空 `src` 销毁**）。卡片只呈现带播放按钮的磨砂遮罩，**用户点击时才动态生成控件**进行流式拉取。 |

#### 2. 移动端接收文件 (桌面端电脑 ➔ 手机浏览器)

| 维度 | 历史遗留逻辑与隐患 | 最新优化实现方案 |
| :--- | :--- | :--- |
| **内存溢出风险** | 用 [downloadFileWithXHR](file:///home/yelon/develop/me/eqrcp/pkg/pages/chat.tmpl.html#L4448-L4477) 强行用 XHR 拉取二进制 Blob 到手机内存，合并后再通过 ObjectURL 模拟下载。下载 200MB 以上大视频时手机浏览器标签页百分百 OOM 崩溃闪退。 | **浏览器原生下载接管**：彻底废弃 XHR 内存缓冲，普通浏览器直接渲染 `<a>` 标签唤起系统自带的下载管理器。数据**直接流式写盘**，JS 堆内存消耗为零，实现绝对零 OOM。 |
| **断点续传能力** | XHR 无法进行分段 Range 请求，断网重新下载。 | **HTTP Range 支持**：Go 服务端在 [handleAttachmentDownload](file:///home/yelon/develop/me/eqrcp/pkg/server/chat.go#L928) 底层采用 `ServeFile` 响应 [ParseRangeHeader](file:///home/yelon/develop/me/eqrcp/pkg/server/progress.go#L19)，手机下载器在断线重试时自动发送 `Range: bytes=N-`，完美实现断点字节续传。 |
| **进度与渲染开销** | 前端主动轮询计算百分比，产生严重的“双重进度条”认知混淆（上传刷一次，下载又刷一次）。 | **服务端驱动进度同步**：Go 服务端通过 [progressResponseWriter](file:///home/yelon/develop/me/eqrcp/pkg/server/util.go#L124) 在传输中实时累加已写字节，通过 SSE 将 `Receiving=true` 和 `Progress` 进度广播。手机端通过接收消息状态在气泡上画出高保真进度条。连接关闭时通过 `defer` 自动复位，清除界面常驻开销。 |

#### 3. 桌面 GUI 客户端发送/接收文件 (桌面端 ➔ 桌面端 / 桌面端 ➔ 手机)

| 维度 | 功能设计 | 内存与传输安全保障 |
| :--- | :--- | :--- |
| **上行 (发送/上传)** | 接收模式下的 TUS 分片上传 | 网页端基于轻量级分片，Go 后端 [handleTusUpload](file:///home/yelon/develop/me/eqrcp/pkg/server/server.go#L2972) 接收分片，物理流式追加合并。内存占用极低。 |
| **下行 (接收/下载)** | Wails 主进程 Go 底层接管 | 桌面端点击“Save”后，Wails 通过 postMessage 发送给 Go 后台接管下载任务：<br>1. **断点探测**：自动探测本地 `.part` 文件的已下载大小，发起 `Range: bytes=localSize-` 头。<br>2. **物理流式追加**：以 `os.O_APPEND` 模式流式写入，内存占用仅为固定的 `32KB` 缓冲区，不堆积大对象。 |

#### 4. 旁路设备接收文件 (中继缓存分发 ➔ 旁路普通客户端/浏览器)

| 维度 | 历史遗留逻辑与隐患 | 最新优化实现方案 |
| :--- | :--- | :--- |
| **大包传输可见性** | 大文件在上传或下载的过程中，多端（包括非相关的旁路设备）都会实时接收到带有蒙版的进度包和文件通知，导致旁路设备的界面上充斥着毫无关联、且不能进行任何交互的垃圾进度条，产生严重的界面视觉骚扰。 | **基于就绪状态的广播过滤**：未下载就绪的临时大文件，在 WebSocket 广播中仅对发送方 A 和桌面端 B 可见，旁路设备 C 在该阶段完全被屏蔽过滤，不感知该消息，保持前台清爽无噪声。 |
| **自动就绪分发机制** | 曾需要桌面端 B 的用户在 GUI 界面上手动点击下载并等待写盘保存，之后才在服务端发起通知解锁旁路。这需要人工频繁干预，无法实现真正的静默会话中继。 | **上传成功即自动分发**：一旦发送端 A 的大文件物理上传并写入桌面端 B 服务端的临时缓冲区（缓存写盘落盘成功），Go 后端在 `handleUpload` 中会自动将其标记为 `Downloaded = true` 并向全网广播。旁路端 C 零人工干预自动展现无蒙版下载气泡（带指向下方的下载箭头），点击即可发起直接下载。 |

### 三、 总结与成效
* **内存安全**：通过**串行上传队列**和**降级为 `<a>` 标签的原生下载机制**，上传和下载两个维度的“大内存缓冲区”全被剥离，保证了不管文件有多大、一次发送多少个，两端设备都永远不会产生内存爆炸。
* **按需加载与播放**：通过 **10MB 预览限制** 与 **流式点播 (Play-on-Demand)**，聊天气泡中没有了常驻渲染的 `<video>` / `<audio>` 播放器和庞大的未激活大图，避免了渲染资源的浪费。只有在用户触发点击时，才从支持 Range 的后端建立流式媒体通道。
* **断点与进度**：由 Go 服务端流式计数并通过 SSE 将真实的“网线内传输字节百分比”同步到气泡上，既获得了高容错的断点续传，又让设备间获得了干净、唯一的进度条显示。
