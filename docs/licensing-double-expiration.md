# EQT 授权双重过期与轻量化授权管理设计方案 (Dual-Expiration & Lightweight License Management)

本文档阐述了 EQT 授权系统中关于“双重有效期”（兑换码截止兑换日 vs 兑换后授权使用时长）的设计实现，以及在无重型用户账户系统下如何解决用户授权丢失与退款销毁的技术优化路线。

---

## 1. 授权双重过期方案设计 (Dual-Expiration Scheme)

为了支持“生成一个兑换码，要求在指定截止日期前兑换，且兑换激活后用户能够获得固定时长（如 1 个月）的授权”这一消费模式，系统需要区分两个不同的时间轴：

1. **最晚兑换期限 (Redeem Expiration - `expires_at`)**：绝对时间戳。超过此日期，此激活码彻底失效，不可再在任何新设备上兑换激活。
2. **授权使用时长 (Activation Duration - `duration_days`)**：相对天数/月数。用户成功激活那一刻起算，离线数字证书（`.lic`）在本地的最大有效天数。

### 1.1 数据库结构升级 (D1 Schema Alteration)
我们在原有的 `licenses` 表中额外引入 `duration_days` 字段：

```sql
-- D1 数据库结构迁移脚本
ALTER TABLE licenses ADD COLUMN duration_days INTEGER DEFAULT NULL;
```

* `expires_at`: 仅存储绝对截止兑换时间（如 `2027-06-25T12:00:00Z`）。
* `duration_days`: 存储兑换激活后可使用天数（如 `30` 代表 1 个月，`365` 代表 1 年，`NULL` / `-1` 代表永久 LIFETIME 授权）。

### 1.2 激活兑换接口动态时效计算 (`src/index.ts` 升级逻辑)
当客户端发起激活请求时，Cloudflare Worker 按如下“第一性原理”重新校验并动态计算证书到期时间：

1. **兑换期限强校验**：
   ```typescript
   if (license.expires_at && license.expires_at !== "LIFETIME") {
     const redeemDeadline = new Date(license.expires_at);
     if (redeemDeadline.getTime() < Date.now()) {
       return new Response(JSON.stringify({ error: "Redeem code has expired and can no longer be activated." }), {
         status: 403,
         headers: { ...corsHeaders, "Content-Type": "application/json" }
       });
     }
   }
   ```
2. **动态到期时间计算**：
   如果该激活码关联了 `duration_days`：
   * **公式**：`finalExpiresAt = Date.now() + (duration_days * 86400 * 1000)`
   * **逻辑**：将该动态生成的绝对时间点写入 Ed25519 签名有效载荷中，颁发给客户端。客户端本地的验签流程与防系统时钟篡改保护将直接继承并执行此时间限制。
    * **旧版回退**：若 `duration_days` 为空，则回退为原有设计（直接继承 `expires_at` 绝对时间点作为最终过期时间，或直接设为 `LIFETIME`）。

### 1.3 压哨激活边界与离线时钟回拨防御
为了解决绝对过期 and 离线环境可能带来的安全漏洞：
* **压哨激活处理 (Late Redemption)**：如果兑换期限在 `12:00` 截止，用户在 `11:59`（仅剩 1 分钟）激活，系统应判定合法，并正常发放完整的 `duration_days` 时长授权（例如获得完整的 30 天使用权）。
* **本地时钟回拨防御**：除基于 `LastTime` 文件的时间倒流校验外，客户端可在后台静默以 10 分钟为步长，向本地混淆文件累加写入“软件激活后的累计运行时间”。一旦该运行时长换算的秒数超出购买授权的总秒数，即便系统时间被回拨至有效期内，软件也会判定过期并锁定付费权限。
* **网络宽限期 (Offline Grace Period) 防灾设计**：客户端每次成功联机校验后，自动在本地续期 `LeaseDuration = 7 days` 租约。在此宽限期内，客户端允许完全处于断网状态使用；当连续 7 天无法联机对账时，才在界面友好提示用户连网以维护激活，以防云端宕机导致断网环境下的软件闪退。

### 1.4 联网对账周期与统一退回免费版机制 (Sync Cycle & Fallback)
为了防范永久断网白嫖并维持架构的单一可信数据源（Single Source of Truth）：
* **联网检测与对账周期**：
  * **应用启动时**：在客户端启动以及日常运行时，在后台启动异步非阻塞协程，自动发起一次联网对账。
  * **对账成功**：如果云端对账成功，更新本地 `LastOnlineSync` 时间，并将 7 天的离线使用宽限期（`LeaseDuration = 7 days`）重新续满。
  * **对账失败（断网/服务器宕机）**：若离上次成功对账时间未超过 7 天，系统自动处于“离线静默信任状态”，所有付费模式正常使用，不打扰用户。一旦连续断网超过 7 天，本地租约失效，客户端将强制降级。
* **统一退回免费版逻辑**：
  * 为防止不同业务模块（如 Share 模式、Receive 模式、Chat 限制等）各自维护一套判定和降级逻辑导致“状态漂移”或漏判，客户端必须确立**统一降级路由**。
  * **统一路由接口**：所有的异常分支（用户手动重置、指纹校验不匹配、证书物理过期、云端退款吊销 Webhook、7 天宽限期到期超限）一旦触发，均禁止自行修改各自模块的状态标志，必须**统一调用 `SetPaidStatus(false, "", "", "")`** 这一全局唯一状态写入入口。
  * **级联广播**：`SetPaidStatus` 通过原子写机制更新本地物理缓存 `chat_usage.json` 及备份文件，并向主应用 `activeSession` 发布“降级/重置系统消息”，界面实时监听该事件并调用热刷新（Hot Reload）通知，使前端 UI 瞬间无缝热切换回免费限制状态，实现状态流转的强一致性。

## 2. 无重型账户系统下的轻量化授权管理优化 (Lightweight Management)

对于消费类软件，长远来看用户需要能自助查询自己的授权、在更换设备或旧设备损坏时手动“解绑”老设备。然而，设计一套包含密码、邮箱验证、个人面板的传统账户系统过重，且破坏了 EQT 极极简、免注册的用户体验。

我们提出以下**无感且安全的轻量化管理方案**：

### 2.1 基于“购买邮箱 + 临时魔链 (Magic Link)”的半自助服务
在用户购买激活码（无论是通过发卡网还是 Stripe）时，购买记录关联用户的“购买邮箱 (Buyer Email)”：

1. **邮箱前置验证**：购买付款前，用户必须在购买页面填写邮箱并接收 6 位随机验证码，用以确认邮箱所有权。
2. **数据库关联**：确认邮箱后，在 `licenses` 表中保存 `buyer_email_hash`（GDPR 隐私加密存储）。
3. **设备自助解绑流程**：
   * 用户在官网帮助中心（`/support`）输入购买时填写的邮箱，点击“获取授权管理链接”。
   * Cloudflare Worker 在后台生成一个有效期 15 分钟的一次性临时 Token 链接（Magic Link）发送到用户邮箱：`https://www.eqt.net.im/manage?token=MAGIC_TOKEN_HEX`。
   * 用户点击邮箱里的链接，该 Token 即刻失效，系统向浏览器下发一个短期（例如 1 小时）的 `HttpOnly` 会话 Cookie，用户无需密码直接以管理员身份进入临时控制台。
   * 在控制台中，用户可以：
     * 查阅该邮箱购买过的所有激活码；
     * 查看各个激活码目前绑定了哪些设备（展现设备操作系统与激活时间）；
     * **自助解绑 (Deactivate)**：对于已经损坏或废弃的老设备，用户点击“解绑”，系统清除 D1 数据库中对应的 `activations` 设备指纹记录，释放 `max_devices` 绑定额度，使用户可在新电脑上重新激活。

### 2.2 基于 Webhook 的自动退款销毁机制
为防范恶意购买后退款继续薅羊毛的行为，可通过第三方支付接口与 Worker 对接实现闭环管理：

* **自动化吊销**：当 Stripe / Gumroad 触发 `refund` 退款事件时，其 Webhook 会向 Worker 发起推送。
* Worker 解析参数，一键更改激活码状态：
  ```sql
  UPDATE licenses SET status = 'revoked' WHERE license_code = ?;
  ```
* 吊销后，由于设备端在每次启动或隔日轮询 `/status` 时都会校验证书有效性，Worker 在判定为 `revoked` 时会阻断验证，促使本地 `.lic` 彻底作废。

### 2.3 购买前置邮箱验证码与支付绑定 (Verification & Checkout Binding)
为了防范发送服务接口被刷频以及邮箱欺骗攻击，系统需具备以下安全设计：
* **人机验证 (Turnstile)**：在点击“发送验证码”前，强制引入 Cloudflare Turnstile 校验，通过后才调用发信 API，杜绝恶意脚本刷爆接口。
* **频率限制 (Rate Limiting)**：使用 Cloudflare KV 的 TTL 功能对单个 IP（60秒一次，每天上限 10 次）和单个邮箱（60秒一次，每天上限 5 次）实行滑动窗口限制。
* **验证码安全匹配**：使用加密安全随机数生成 6 位验证码，将 `sha256(email + code + salt)` 缓存于 KV 中（10分钟过期）。只要尝试匹配失败 3 次或验证成功，立即从 KV 删除，防止暴力破解。
* **支付会话 (Checkout) 绑定锁死**：验证成功后，Worker 创建支付网关 Checkout 订单时，将已验证的邮箱强行注入为 `customer_email` 且设为**只读不可修改**，防止用户在收银台恶意替换成其他未验证的邮箱。

### 2.4 控制台安全与下推吊销 (Magic Link Console Security)
* **解绑频次限额**：为防止通过“多设备轮转”机制逃避设备绑定数限制，每个授权码**每月最多仅允许解绑 2-3 次**。
* **下推吊销同步 (Push-Deactivation)**：用户在网页解绑某台离线老设备后，客户端在联网检测或每日自动 `/status` 对账时，一旦发现本地指纹不在云端合法授权名单里，会自动擦除本地 `.lic`，退回免费模式，避免离线设备无限期白嫖。

### 2.5 GDPR 隐私保护与哈希存储 (GDPR & Data Hashing)
* 为了防范云端 D1 数据库泄露导致的用户邮箱曝光风险，在 `licenses` 表中不直接存储 `buyer_email` 明文，转而存储哈希值：`buyer_email_hash = sha256(buyer_email.toLowerCase())`。
* 用户在管理后台输入邮箱时，系统在内存中对其转小写并计算哈希值，以此与 D1 中的哈希记录安全匹配。

### 2.6 本地落盘文件物理隔离与加密混淆设计规范 (State Files Isolation & Encryption)
为确保在本地运行时客户端的状态一致性、隐私安全以及高并发防损坏，对四个核心落盘文件建立物理隔离和分级加密规范：
1. **物理文件隔离设计（不可合并）**：
   * **`config.yml` (系统全局配置，明文，极低频写)**：保存全局网络及基础行为设置（端口、路径等）。必须保持明文，允许用户在界面异常或端口冲突时使用普通文本编辑器手动配置。
   * **`license.lic` (离线许可证，明文+数字签名，极低频写)**：核心防线是云端 Ed25519 的不可伪造签名。需保持明文形式以加速程序启动时的反序列化和验签效率，无需二次加密。
   * **`chat_usage.json` (使用量限制，高频写，强混淆防篡改)**：保存高级付费标志与运行时长、时钟校验数据。因为高频写入且用户极易通过篡改数值白嫖，必须进行物理隔离，防止阻塞主配置读写。
   * **`desktop-agent-history.json` (传输历史，追加写，加密防泄露)**：体积会随着历史使用不断追加膨胀。需与其物理隔离，避免高频写时间限制时产生不必要的全量大文件 I/O 损耗。
2. **加密混淆安全分级**：
   * **使用量防篡改级别**：对 `chat_usage.json` 及其隐藏双重防线 `.eqt_sys_state` 必须使用对称 XOR 混淆（`xorObfuscate`）加密，阻断用户通过本地编辑文本修改使用时间及付费标记的意图。
   * **历史记录隐私保护级别**：对 `desktop-agent-history.json` 引入基于设备硬件指纹派生出的对称密钥执行轻量化 XOR 混淆，防止本地其他恶意软件读取传输历史，保护 GDPR 用户隐私。
