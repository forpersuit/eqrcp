## 2. 硬件设备指纹生成（防重装系统）

在不依赖用户注册账号的场景下，为了确保“一台设备”在重装系统后依然能正确识别，必须采集不可变的物理硬件信息。

### 2.1 采集指标与加权方案
单纯收集某一个硬件 ID（如 MAC 地址）是不安全的，因为网卡可禁用、MAC 可随机，且重装系统后虚拟网卡顺序会变。因此，采用 **加权硬件指纹（Weighted Hardware Fingerprint）** 算法：

1. **主板UUID (Motherboard UUID)** - 权重 40%
   * Windows: WMI 查询 `Win32_ComputerSystemProduct` 的 `UUID`。
   * Linux: 读取 `/sys/class/dmi/id/product_uuid`。
   * 特点：重装系统绝对不变，虚拟机中也稳定（但多台同模板虚拟机可能相同）。
2. **CPU 序列号 (CPU Serial Number)** - 权重 30%
   * 通过汇编指令 `cpuid` 读取。
   * 特点：硬件固有，物理机重装系统不变。
3. **系统物理硬盘序列号 (Physical Disk Serial)** - 权重 30%
   * Windows: WMI 查询主系统盘 `Win32_PhysicalMedia` 的 `SerialNumber`（非分区卷标 ID，卷标格式化会变）。
   * Linux: 读取 `/sys/block/sda/device/serial` 或使用 `udevadm`。
   * 特点：物理硬盘固有，格式化或重装系统不变，但用户更换硬盘时会改变。

### 2.2 宽松匹配机制（容错）
为了解决用户正常“更换固态硬盘”或“升级CPU”后导致激活失效的痛点，设备码生成不使用简单的 `Hash(UUID + CPU + Disk)`，而使用 **多部分比对规则**：
* 收集并保存三个硬件特征：`[UUID, CPUSn, DiskSn]`。
* 当本地计算的硬件特征与激活信息对比时，**只要有任意 2 项一致**，即判定为同一台设备，并自动在本地更新已改变的第 3 项特征。

## 3. 买断制防任意分发与破解分析

买断制最大的痛点在于：**用户购买一次后，将授权文件或激活过的 EXE 直接分享给其他人。**

### 3.1 方案：直接将机器码写死到 EXE 中
* **思路**：用户购买后，服务器编译一份专属该用户机器码的 `eqt.exe` 供其下载。
* **安全性分析（为什么容易被破）**：
  * **静态修改（Patching）**：逆向破解者只需要使用工具（如 IDA Pro / Ghidra），找到验证机器码的跳转指令（如 `jne` 汇编指令），将其直接修改为 `nop` 或强行修改为 `je`。哪怕写死机器码，只要比对逻辑在本地被短路，该 EXE 就会变成“通杀版”。
  * **动态劫持（Hooking/MitM）**：如果 EXE 联网验证，破解者可以通过修改 `hosts` 文件，将验证域名指向本地 `127.0.0.1` 伪造验证通过的 JSON 响应。

### 3.2 升级防分发方案：非对称数字签名授权（.lic 证书）
为了避免为每个用户编译专属二进制的极高成本，以及防止本地写死被轻易 Patch，建议引入 **非对称加密授权文件（License File）** 机制：

1. **证书生成流程**：
   * 客户端本地获取硬件指纹 `DeviceID = SHA256(UUID + CPU)`。
   * 购买时，用户将 `DeviceID` 提交给官方。
   * 官方服务器使用 **私钥（Private Key）** 对 `DeviceID + LicenseType + ExpiryDate` 进行签名，生成 `Signature`，打包为 `.lic` 授权文件给用户。
2. **本地校验流程**：
   * 客户端安装 `.lic` 后，程序启动时使用内置的 **公钥（Public Key）** 验签。
   * 如果验签通过，再比对签名中的 `DeviceID` 是否与当前运行机器的硬件指纹一致。
   * **安全性**：逆向者没有私钥，无法制作出“注册机（Keygen）”。

### 3.3 终极防逆向 Patch 策略：隐式依赖与代码混淆

如果破解者直接把本地 `VerifyPublicKey()` 的返回值强行改成 `true`，上述非对称签名依然可能被绕过。因此必须使用以下手段提高防线：

1. **核心逻辑散落验证（No Single Point of Failure）**：
   * 绝对不要设计类似 `if (!checkLicense()) { exit(); }` 的单点判断。
   * 应该把 `DeviceID` 作为**解密密钥的一部分**。例如，局域网聊天所需的某些静态 HTML/JS 资源、默认的加密随机盐等，其解密 Key 隐式地依赖于合法的证书签名。
   * 如果授权签名被强行 Patch 绕过，但因为缺少真正的解密密钥，程序运行到传输或聊天渲染时，会由于数据解密失败而崩溃，从而让只改跳转的“粗暴破解”失效。
2. **Go 代码混淆编译（Garble）**：
   * Go 默认包含大量的符号信息和反射元数据。
   * 生产发布时，必须使用 [garble](https://github.com/burrowers/garble) 对 Go 源码进行打包混淆。混淆后，函数名、包名、变量名均被替换为随机无意义字符，极大地增加逆向分析的难度。

## 4. 联机与脱机混合机制

### 4.1 核心：30 天脱机使用额度校验
为了防止用户购买年度订阅后通过“断网+倒拨时钟”无限期白嫖：
* 当设备连网时，后端会下发一个经过私钥签名的最新网络时间戳包 `TimeToken`。
* 即使客户端不联网，只要它在 30 天内启动运行，就能够独立验证 `TimeToken` 并在本地完成授权解密。
* 一旦断网超过 30 天，由于本地时间已远大于签名时间加上 30 天有效期，程序自动锁死，要求连网刷新。
* 本地使用隐藏备份文件中的 `LastTime` 监控本地时间倒拨。

### 4.2 买断制设备限制与公网泄露阻击
* **多设备激活限制（例如每码限 3 台设备）**：
  * 买断版用户初次联网激活时，在官方服务器注册设备码 `DeviceID`，服务器计数 +1。
  * 超过限制则拒绝签发新设备的 `.lic`。
* **公网泄露阻击（CRL 吊销列表）**：
  * 虽然买断制支持离线使用，但一旦该设备连网，程序会在后台静默请求服务器的 CRL（证书吊销列表）服务。
  * 如果官方发现某个买断激活码在公网上同时有几十台不同的 `DeviceID` 在发出请求，说明该激活码已被公开分享。服务器会将该激活码拉黑并写入 CRL。
  * 联网的客户端获取到最新的 CRL 后，会在本地执行拉黑，废除该授权文件的合法性。

## 5. 常见破解路径与防御对策汇总

| 攻击路径 | 破解手法描述 | EQT 针对性防御方案 |
| :--- | :--- | :--- |
| **时钟回拨** | 修改电脑系统时间回到订阅期内 | **时钟抗回拨锁**：记录 `LastTime`，发现当前时间小于 `LastTime` 直接永久死锁，需重置授权或连网同步方可解锁。 |
| **网络劫持** | 修改 hosts 或使用 Fiddler 劫持域名验证 | **SSL Pinning & 签名校验**：网络传输使用 HTTPS 且硬编码证书指纹；所有网络回复均需经过服务器私钥的非对称签名，即使劫持也无法伪造签名。 |
| **二进制打补丁** | 使用 Hex 编辑器或传统 IDA Patch 关键跳转（如 `jne` 变 `je`） | **散落式非对称解密**：将设备特征码作为加密算法的因子，使破解者不仅需要改跳转，还要保证解密密钥正确，大幅提高分析门槛；使用 `garble` 编译剔除所有可见的方法名称。 |
| **激活码泄露** | 一个买断激活码被发到贴吧/论坛上全网共享 | **设备并发监控与 CRL 吊销机制**：联网时静默校验设备指纹并发数，超出阈值直接远程注销并在吊销列表中发布，阻断其继续传播。 |
